# 1. Purpose

This document defines the quality requirements for every software artifact produced within the Wealth Platform.

Its objective is to ensure that every implementation generated by human developers or AI coding agents satisfies the same quality standards.

Quality is treated as an implementation requirement rather than a post-development activity.

---

# 2. Quality Philosophy

The platform follows these principles:

* Quality is designed, not inspected.
* Every business rule must be verifiable.
* Every implementation must include its validation.
* AI-generated code must satisfy the same standards as human-written code.
* Functional Specifications are the primary source for quality validation.

---

# 3. Definition of Done

A feature is considered complete only when all of the following artifacts exist:

* Production code.
* Unit tests.
* Integration tests.
* API contract (if applicable).
* Database migration (if applicable).
* Frontend implementation (if applicable).
* Documentation updates.
* Static analysis passes.
* Build succeeds.
* Acceptance Criteria are satisfied.

Code alone is never considered complete.

---

# 4. Test Pyramid

The preferred testing strategy is:

```text
                Manual Validation
                       ▲
                End-to-End Tests
                       ▲
              Integration Tests
                       ▲
                  Unit Tests
```

Most tests should be automated.

Manual testing focuses on user validation rather than regression detection.

---

# 5. Required Automated Tests

Every feature must include, where applicable:

## Unit Tests

Validate:

* business rules;
* domain services;
* value objects;
* calculations;
* validators.

---

## Integration Tests

Validate:

* persistence;
* repositories;
* application services;
* module interaction.

---

## API Tests

Validate:

* endpoints;
* request validation;
* authorization;
* response contracts.

---

## UI Tests

Validate:

* critical user journeys;
* navigation;
* forms;
* business workflows.

Playwright is the preferred framework.

---

# 6. Manual Validation

Manual testing is limited to:

* User Acceptance Testing (UAT).
* Exploratory Testing.
* UX validation.
* Visual verification.

Manual regression testing should be avoided.

---

# 7. Traceability

Every Acceptance Criterion defined in a Functional Specification must be traceable to one or more automated tests.

The relationship should be:

```text
Functional Specification
        ↓
Acceptance Criteria
        ↓
Automated Tests
```

No business rule should exist without automated validation.

---

# 8. AI Quality Requirements

Every AI coding agent implementing this project must:

* generate tests together with production code;
* never postpone testing;
* never generate placeholder tests;
* never skip failed tests;
* update tests when business rules change.

Generated tests must be deterministic.

---

# 9. Static Quality Gates

Every change must satisfy:

* successful build;
* formatting rules;
* linting rules;
* static analysis;
* security analysis;
* dependency validation.

No pull request should bypass these gates.

---

# 10. Code Coverage

The project does not define a minimum percentage.

Instead, the objective is:

> Every business rule must be covered by automated tests.

Coverage is evaluated by business value rather than raw percentages.

---

# 11. Performance Validation

Critical operations must include performance validation.

Examples:

* financial calculations;
* portfolio aggregation;
* provider synchronization;
* dashboard generation.

Performance regressions should be automatically detected where practical.

---

# 12. Security Validation

Security-sensitive functionality must include automated validation.

Examples:

* authentication;
* authorization;
* permission checks;
* tenant isolation;
* audit logging.

---

# 13. Financial Validation

Financial calculations require additional guarantees.

Every financial calculation must be:

* deterministic;
* reproducible;
* independently testable;
* free of hidden side effects.

Identical inputs must always produce identical outputs.

---

# 14. AI Artifact Requirements

A completed implementation generated by AI must produce:

* production code;
* automated tests;
* API documentation updates;
* database migrations;
* OpenAPI updates (if applicable);
* frontend changes (if applicable).

Partial implementations are not acceptable.

---

# 15. Continuous Integration

The CI pipeline must verify:

1. Build.
2. Static analysis.
3. Unit tests.
4. Integration tests.
5. API tests.
6. UI tests (where applicable).
7. Security validation.
8. Packaging.

Deployment is allowed only if all mandatory quality gates succeed.

---

# 16. Implementation Contract

## Responsibilities

This document defines the mandatory quality requirements for every implementation.

## Required Deliverables

Every feature implementation must include:

* executable code;
* automated validation;
* documentation updates;
* deployment compatibility.

## Forbidden Practices

The following are prohibited:

* code without tests;
* disabled tests;
* ignored failures;
* manual regression testing as the primary validation mechanism;
* financial calculations without deterministic tests.

---

# 17. AI Implementation Notes

AI coding agents must interpret this document as mandatory.

For every implemented Functional Specification they must generate:

* backend implementation;
* frontend implementation (when applicable);
* unit tests;
* integration tests;
* API tests;
* UI tests for critical user journeys;
* required documentation updates.

The absence of any mandatory artifact means the implementation is incomplete.

---

# 18. Acceptance Criteria

This document is complete when:

* quality expectations are explicit;
* automated validation is mandatory;
* manual testing responsibilities are limited;
* AI quality obligations are documented;
* every implementation artifact is defined.
