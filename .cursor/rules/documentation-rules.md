# DOCUMENTATION RULES

These rules define how documentation must be written.

## Document Status

Use one of:

- Draft
- Review
- Accepted
- Deprecated

Only Accepted documents may be treated as authoritative.

## Standard Metadata

Every official document should include:

- Document ID
- Version
- Status
- Owner
- Depends On
- Required By
- Change Log

## Capability Document Structure

Capability specifications should include:

1. Purpose
2. Business Problem
3. Business Value
4. Scope
5. Responsibilities
6. Core Concepts
7. Domain Principles
8. Lifecycle
9. Business Rules
10. Actors
11. Domain Events
12. Dependencies
13. Permissions
14. AI Considerations
15. Acceptance Criteria
16. Open Questions
17. Future Evolution
18. Change Log

## Writing Style

- Use English for official documentation.
- Be precise.
- Prefer short sentences.
- Avoid vague words such as "maybe", "probably", "somehow".
- Avoid implementation details unless the document is explicitly technical.

## Mermaid

Use Mermaid diagrams when they clarify relationships or flows.
